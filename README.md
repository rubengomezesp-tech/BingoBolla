# BingoBolla v3 — Real product

## What's new in v3

This drops the "demo" mindset. Everything is real now:

✓ **Auth** via Supabase magic link → `/login`
✓ **Lobby** at `/lobby` reads live rooms from DB
✓ **Live game rooms** at `/room/[id]` with realtime balls, multi-card support, chat, claim bingo
✓ **Server-authoritative gameplay** — cards generated in Postgres, wins verified in Postgres
✓ **Atomic coin transactions** — RPC `buy_ticket` and `claim_bingo` with proper locks
✓ **Provably fair** — seed hash committed before round, revealed after
✓ **Caller worker** — Node script that drops balls every N seconds for active games

## Apply migration

Rotate your DB password first (you leaked it), then:

```bash
cd ~/bingobolla
tar -xzf ~/Downloads/bingobolla-v3.tar.gz

# Apply schema to Supabase (replace NEW_PASSWORD with your rotated password)
psql "postgresql://postgres:NEW_PASSWORD@db.atfsgvetqxjmmsokswja.supabase.co:5432/postgres" \
  -f supabase/migrations/001_init.sql

# Add SUPABASE_SERVICE_ROLE_KEY to .env.local (get from Supabase Dashboard → Settings → API → service_role)
# (already have NEXT_PUBLIC_SUPABASE_URL and NEXT_PUBLIC_SUPABASE_ANON_KEY)

# Install dotenv for the caller worker
npm install --save-dev dotenv

# Start everything in 2 terminals:
npm run dev                    # terminal 1 — Next app
node scripts/caller.mjs        # terminal 2 — ball caller

# Open http://localhost:3000
# Sign up → check email for magic link → /lobby → pick room → buy ticket → wait for caller to start dropping balls
```

## Architecture notes

**Why server-authoritative?** Bingo is trivially cheatable client-side. Anyone with devtools could fake a win. So:
- Cards generated by `generate_bingo75_card()` Postgres function, hash stored in `cards.card_data`
- Balls only dropped by the caller worker via `call_next_ball(game_id)` RPC
- Wins verified by `claim_bingo(card_id, pattern)` which recomputes from server-side state
- Client UI is only for display

**Why a separate caller worker?** Supabase doesn't have native sub-minute crons on free tier. Run it locally during dev (`node scripts/caller.mjs`), deploy to Railway/Fly.io for production. It's stateless — just polls active games and drops balls per room's `ball_interval_ms`.
